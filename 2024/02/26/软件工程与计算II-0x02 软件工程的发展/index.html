软件工程与计算II-0x02 软件工程的发展

发表于 2024-02-26 分类于 软件工程与计算II

2.1 软件工程发展脉络

年代 总结
20 世纪 50 年代 科学计算。 以机器为中心编程。 像生产硬件一样生产软件。
20 世纪 60 年代 业务应用。 软件不同于硬件。 用软件工艺的方式生产软件。
20 世纪 70 年代 结构化方法。 瀑布模型。 强调规则和纪律。 奠定了软件工程的基础。
20 世纪 80 年代 追求生产力最大化。 现代结构方法/面向对象编程广泛应用。 重视过程的作用。
20 世纪 90 年代 企业为中心的大规模软件系统开发。 追求快速开发、可变更新和用户价值。 Web 应用出现。
21 世纪 00 年代 大规模 Web 应用。 大量面向大众的 Web 产品。 追求快速开发、可变更性、用户价值和创新。

2.2 20 世纪 50 年代的软件工程

1. 基础环境
  1. 虚拟计算机：研究用大型机。

​ 数量有限，编写程序基于基本输入输出系统 (Basic Input Output System, BIOS)，甚至直接基于硬件。编译环境只有汇编语言的编译器。

  1. 主要抽象软件实体：依赖于硬件的软件。

​ 一个很大进步是汇编语言出现和普及，其隐藏了机器特定细节，将程序员从一些无关紧要的问题中解放出来。

​ 20 世纪 50 年代后期开始出现 Fortran 等面向语句编程的初级语言。

2. 主要现实问题

​ 计算资源宝贵，软件以机器为中心 (machine-centric)。

3. 软件开发方法与技术

​ 暂未出现，精力集中于硬件上。

4. 软件开发过程

​ “制造软件和硬件是一样的”。

5. 重要思想

​ 重视产品质量，进行评审和测试。

6. 总结
  • 科学计算。
  • 以机器为中心编程。
  • 像生产硬件一样生产软件。

2.3 20 世纪 60 年代的软件工程

1. 基础环境
  1. 虚拟计算机：商业大型机。

​ 价格昂贵。为提高开发效率，企业愿意购买协助开发的系统软件，如操作系统、高级语言编译器、命令行模式的开发环境、代码工具库等。

  1. 主要抽象软件实体：第 3 代语言。

​ 提供了函数（过程）机制，因为编译器已经能够很好完成对语句的优化，程序员可以转移出精力考虑如何从更高层次的函数（过程）构建程序。

​ 提供了更好类型系统和数据结构化。

2. 主要现实问题

​ 需要软件解决的问题从科学计算转向业务应用，称作以应用为中心 (application-centric)。

​ 业务应用开发比科学计算复杂得多，因为其问题和解决逻辑都是复杂且模糊的。

3. 软件开发方法与技术

​ 总体上依靠程序员能力，“工艺式”开发。

4. 软件开发过程

​ “工艺式”程序员使用“构建-修复” (build-fix) 过程开发软件。

​ 导致了“个人英雄主义编程”，“牛仔程序员”是常见角色。为了赶在 ddl 前完工，召集夜猫子 debug，然后被当作英雄。

5. 重要思想
  1. 软件不同于硬件。
  • 软件与现实关系更加密切，对需求规格化更难。
  • 软件比硬件容易修改得多，并且不需要昂贵的生产线复制产品。
  • 软件没有损耗。
  • 软件不可见。
  1. 避免“工艺式”生产，用工程方式生产软件
  • ”工艺式“软件生产导致了质量低、开发效率低、成本提高、可维护性低、没有适当的文档材料、对开发成本和进度难以估计等问题。
  • 1968 年和 1969 年召开两届里程碑式的“软件工程”会议。1968 年的会议提出了“软件危机”的说法，1969 年的会议指出“软件工程”方向，用工程的方法生产软件。
6. 总结
  • 业务应用。
  • 软件不同于硬件。
  • 用软件工艺的方式生产软件。

2.4 20 世纪 70 年代的软件工程

1. 基础环境
  1. 虚拟计算机：商业微型机。

​ 体积更小，成本更低。

​ 系统软件方面，关系代数和演算的成熟产生了数据库管理系统 DBMS 软件，极大提高开发者处理可持久化数据能力。

  1. 主要抽象软件实体：结构化程序设计理论。

​ 结构化程序设计理论使用函数（过程）、块结构 (block structure) 和三种基本控制结构（顺序、循环、分支）为基础构建程序。

​ 最大粒度软件抽象实体仍是函数（过程）。

2. 主要现实问题

​ 更多的企业带来更多的业务应用。对软件产品需求增多。

​ 结构化程序设计理论使得小型程序开发成功率有了一定保证。保证复杂软件系统开发的成功、解决复杂软件系统的复杂性成为开发工作主要指导思想。

3. 软件开发方法与技术

​ 70 年代早期开始广泛使用结构化编程方法，使用函数（过程）构建程序，使用块结构和三种基本控制结构组织函数（过程）的代码，使用程序流程图描述逻辑进行程序设计，使用逐步精细化、自顶向下的软件开发方法进行开发。

​ 70 年代中后期，结构化方法从编程活动扩展到分析和设计活动，围绕功能分解思想和层次模块结构，使用数据流图 (Data Flow Diagram, DFD)、实体关系图 (Entity Relation ship Diagram, ERD) 和结构图 (Structure Chart)，建立了结构化设计、结构化分析、 JSP (Jackson Structured Programming) 等结构化分析与设计方法。

4. 软件开发过程

​ “瀑布模型”：开发中越早发现缺陷并修复，工作量和成本越低，因此强调一个结束阶段进行验证和检验，尽早修复缺陷，降低成本，提高成功率。

​ 但瀑布模型被解释为一个纯顺序过程，这种僵硬的顺序要求不符合软件开发实际。

5. 重要思想

​ 越早发现和修复问题，代价越低。

6. 总结
  • 结构化方法。
  • 瀑布模型。
  • 强调规则和纪律。
  • 奠定了软件工程的基础。

2.5 20 世纪 80 年代的软件工程

1. 基础环境
  1. 虚拟计算机：个人计算机。

​ 图形用户界面 GUI 出现。

  1. 主要抽象软件实体：面向对象编程方法。

​ 核心抽象实体是“对象/类”，粒度更大。

2. 主要现实问题

​ 业务应用。

3. 软件开发方法与技术
  1. 结构化方法

​ 产生现代结构化方法，如信息工程、现代结构化分析。更加注重系统构建而非程序构建。

  1. 面向对象编程

​ C++ 的出现和应用。C++ 的成果是因为其保留了 C 的各种特性，另一方面是由于面向对象语言支持复用和更适于复杂软件开发。可复用性，可修改性。

  1. 软件复用
4. 软件开发过程
  1. 过程模型

​ ①迭代式开发；②并行开发，最大化缩减整体开发时间；③快速开发，利用原型等方法缩短开发时间。

  1. 过程评价

  2. 使用工具支持软件开发过程

​ 支持需求分析与软件设计的建模工具、支持软件测试的测试工具、支持编程的集成开发环境、支持团队协作的配置管理工具等。

5. 重要思想

​ 没有银弹。 复杂度、一致性、可变性、不可见性。软件内在特性决定复杂软件系统开发的困难是：充分利用多种模型手段，尽可能将内部元素与外部接口可视化，以推理其正确性、可靠性、完备性和可修改性。

​ 重视人的作用。人是项目最大的资产。卓越设计人员是“银弹”之一。“个人与交互胜过过程与工具”。

6. 总结
  • 追求生产力最大化。
  • 现代结构方法/面向对象编程广泛应用。
  • 重视过程的作用。

2.6 20 世纪 90 年代的软件工程

1. 基础环境

  1. 虚拟计算机：Internet。

  2. 主要抽象软件实体：面向对象分析与设计方法、软件体系结构方法

2. 主要现实问题

​ 业务应用。复杂度、可修改性、开发周期、用户价值。

​ 维护人员数量开始超过开发人员数量。

3. 软件开发方法与技术

​ 面向对象方法。

​ 软件体系结构。

​ 人机交互：基本目标是开发更加友好的软件产品，最低标准是普通人使用时比较顺畅，较高标准是在使用时感到满足和愉悦。

​ 需求工程：利用工程化手段进行需求处理，以保证需求处理正确运行。

​ 基于软件复用的大规模软件系统开发技术：框架 (frame) 和构件 (component)。

​ Web 开发技术：Web 程序的数据描述标准 XML。

4. 软件开发过程
  1. 过程模型

​ 主流的重量级过程方法 RUP (Rational Unified Process) 和很多轻量级过程方法。

​ 前者思想是在一个基本框架下组织和使用最佳实践方法，有全程的软件过程工具支持。

​ 后者则是抛弃传统上僵化的过程框架，选择一套能够互相配合的最佳实践方法组织开发过程，21 世纪前 10 年发展为敏捷 (agile) 方法。

  1. 过程改进

  2. 开源软件

​ 一个公司将自己产品开源而不牟利通常有两个理由：打击竞争对手，将与对手相似产品开源；促进与开源相关其它产品的销售。

5. 重要思想

​ 重视最佳实践方法。最佳实践是指那些在软件开发实践中被反复证明能减少开发时间、降低开发成本、提高产品质量的实践方法。

6. 总结
  • 企业为中心的大规模软件系统开发。
  • 追求快速开发、可变更新和用户价值。
  • Web 应用出现。

2.7 21 世纪 00 年代的软件工程

1. 基础环境

  1. 虚拟计算机：更广 (Internet) 和更小 （嵌入式设备与移动终端）。

  2. 主要抽象软件实体：对面向对象的一些概念做了更严谨的规格化，发布了 UML2.0。

2. 主要现实问题
  • 基于 Internet 而非 Intranet 的大规模 Web 应用日益成为主流。
  • 面向消费大众的软件产品需求出现爆炸性增长。
3. 软件开发方法与技术
  1. 延续 90 年代技术

  • 软件体系结构。

  • 人机交互

  • 需求工程

  • 基于软件复用的大规模软件系统开发技术

  1. Web 技术发展

  2. 特定领域的软件工程方法

4. 软件开发过程

​ 敏捷方法继续发展，产生广泛影响。

​ 探索外包软件的开发方式。

6. 总结
  • 大规模 Web 应用。
  • 大量面向大众的 Web 产品。
  • 追求快速开发、可变更性、用户价值和创新。
