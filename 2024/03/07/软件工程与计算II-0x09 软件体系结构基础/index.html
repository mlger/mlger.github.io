软件工程与计算II-0x09 软件体系结构基础

发表于 2024-03-07 分类于 软件工程与计算II

9.2 理解软件体系结构

9.2.1 定义

​ 软件体系结构 = { 部件 (component)，连接件 (connector)，配置 (configuration) }

  • "部件"是基本组成单位之一，承载系统主要功能，包括处理与数据。
  • “连接件”定义了部件间的交互，是连接的抽象表示。
  • “配置”定义了“部件”以及“连接件"之间的关联方式，将它们组织成系统的计算机部件与部件之间的交互。

​ 一个软件系统的体系结构规定了系统的计算部件和部件之间的交互。

​ 连接件是一个与部件平等的单位。部件与连接件是比类、模块等软件单位更高层次的抽象。

9.2.2 区分软件体系结构的抽象与实现

​ 以部件、连接件和配置为基本单位组织的模型就是软件体系结构的抽象，基本目的是描述软件系统的整体功能和组织，不涉及程序设计语言提供的各种编程机制。

​ 软件体系结构设计是先使用抽象机制完成软件系统的总体功能部署，然后再将抽象模型等价转化为实现模型。

​

9.2.3 部件

​ 部件可以分为原始 (primitive) 和复合 (composite) 两种类型。原始类型的部件可以直接被实现为相应的软件实现机制，具体实现粒度要依部件的复杂度而定。复合类型的部件则由更细粒度的部件和连接件组成。

​

9.2.4 连接件

​ 连接件也分为原始和符合两种类型。原始类型的连接件可以直接被实现为相应的软件实现机制。复合类型的连接件则由更细粒度的部件和连接件组成。

​

​ 连接件的软件实现可以分为隐式和显式两种。隐式连接件不需要专门开发，附属在部件的实现之上，部件实现完成，连接件实现自然也就完成。显示类型机制通常需要进行一些复杂的逻辑处理，需要开发者进行专门的实现。

9.2.5 配置

​ 配置是一种机制，专门将部件和连接件整合起来，构成系统的整体结构，达到系统的设计目标。

​ 体系结构语言 (Architecture Description Language, ADL)，用于描述软件体系结构的形式化模型语言。

​ 利用配置连接相互独立的部件和连接件，由如下好处：

  • 可以实现部件和连接件的一次定义，多次使用；
  • 在具体交互中，参与的部件不再固定，可以随时发生变化；
  • 对于具体部件而言，参与的交互也不再固定，可以随时参与或退出一个交互。

​

9.3 体系结构风格初步

耦合性（英语：Coupling）或称耦合力或耦合度，是一种软件度量，是指一程序中，模块及模块之间信息或参数依赖的程度。

内聚性（英语：Cohesion）是一个和耦合性相对的概念，一般而言低耦合性代表高内聚性，反之亦然。耦合性和内聚性都是由提出结构化设计概念的赖瑞·康斯坦丁所提出[1]。低耦合性是结构良好程序的特性，低耦合性程序的可读性及可维护性会比较好。

9.3.1 主程序 / 子程序
  1. 简要描述

    层次结构。

    设计决策与约束有：

    • 基于声明 - 使用（程序调用）关系建立连接件，以层次分解方式建立系统部件构成层次结构。
    • 每个上层部件可以“使用”下层部件。
    • 系统应该是单线程执行。
    • 子程序只能够通过上层转移来获得控制权。

  2. 实现

    主要实现机制是模块实现，主程序 / 子程序风格是基于部件与连接件建立的高层结构，它的部件不同于程序，而是更加粗粒度的模块。

  3. 效果

    优点：

    • 流程清晰，易于理解。
    • 强控制性。

    缺点：

    • 强耦合，难以修改和复用。
    • 限制了各部件间的数据交互，可能使得不同部件使用隐含的共享数据交流，产生不必要的公共耦合，破坏“正确性”控制能力。

  4. 应用

    主要用于能够将系统功能依层次分解为多个顺序执行步骤的系统。

9.3.2 面向对象式

  1. 简要描述

    该风格将系统组织为多个独立的对象，每个对象封装其内部的数据，并基于数据对外提供服务。不同对象之间通过协作机制共同完成系统任务。

    设计决策与约束：

    • 使用信息内聚达到标准为系统建立对象部件。每个对象对外提供服务接口，并隐藏内部数据表示。
    • 基于方法调用机制建立连接件，将对象连接起来。
    • 每个对象负责维护其自身数据的一致性和完整性，并以此为基础对外提供“正确”的服务。
    • 每个对象都是一个自治单位，不同对象之间是平级的，没有主次、层次、分解等关系。

  2. 实现

    它的“对象”是部件，属于高层结构的元素，但不同于面向对象分析方法中的“对象”实体。主要实现机制是模块实现。

  3. 效果

    优点：

    • 内部实现的可修改性。
    • 易开发、易理解、易复用的结构组织。

    缺点：

    • 接口的耦合性。
    • 标识 (identity) 的耦合性，一个对象要与其它对象交互，就必须知道其它对象的标识。
    • 副作用：对象的重入 (reentry) 问题：A.f() 调用了 B.p()，B.p() 又调用了 A.q()，就有可能导致 A.q() 失败。

  4. 应用

    适用于哪些能够基于数据信息分解和组织的系统软件。

9.3.3 分层

  1. 简要描述

    分层风格根据不同的抽象层次，将系统组织为层次式结构。每个层次被建立为一个部件，不同部件之间通常用程序调用的方式连接。

    决策与约束有：

    • 最底层到最高层，部件的抽象层次逐渐提升，
    • 两个层次之间的连接要遵守特定的交互协议，该协议是成熟、稳定和标准化的。
    • 跨层次的连接是禁止的。
    • 逆向连接是禁止的。

  2. 实现

    分层风格通常是不限粒度的模块实现。每个层次部件可以表示为一个包，包内含有它的所有实现模块。

  3. 效果

    优点：

    • 设计机制清晰，易于理解。
    • 支持并行开发。
    • 更好的可复用性与内部可修改性。

    缺点：

    • 交互协议难以修改。
    • 性能损失。禁止跨层调用，带来冗余调用处理，致使性能损失。
    • 难以确定层次数量和粒度。

  4. 应用

    适用于如下特点系统：

    • 主要功能是能够在不同抽象层次上进行任务分解的复杂处理。
    • 能够建立不同抽象层次之间的稳定交互协议。
    • 没有很高的实时性需求，能够容忍稍许的延迟。

    如 TCP/IP 网络模型，ISO 网络通信模型。

9.3.4 MVC

  1. 名称

    模型 - 视图 - 控制 (Model-View-Control, MVC) 风格。

  2. 简要描述

    以程序调用为连接件，将系统功能组织为模型、视图和控制三个部件。

    模型封装了系统数据和状态信息，实现业务逻辑，对外提供数据服务和执行业务逻辑。

    视图封装了用户交互，提供业务展现，接收用户行为。

    控制封装了系统控制逻辑，根据用户行为调用需要执行的业务逻辑和数据更新，并且根据执行后的系统状态决定后续的业务展现。

    设计决策与约束有：

    • 模型、视图、控制是关于业务逻辑、表现和控制的三种不同内容抽象。
    • 如果视图需要持续地显示某个数据状态，那么它首席需要在模型中注册1对该数据的兴趣。
    • 视图只能使用模型的数据查询服务，只有控制部件可以调用可能修改模型状态的程序。
    • 用户行为虽由视图发起，但必须转交给控制部件处理。
    • 模型部件相对独立，既不依赖于视图，也不依赖于控制。

  3. 实现

    需要为模型、视图、控制每个部件实例建立模块实现，各模块存在导入/导出关系，程序调用连接件不需要显式实现。

  4. 效果

    优点：

    • 易开发性。
    • 视图和控制的可修改性。
    • 适宜于网络系统开发的特征。

    缺点：

    • 复杂性。
    • 模型修改困难。

  5. 应用

    主要用于网络系统开发。
