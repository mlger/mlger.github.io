软件工程与计算II-0x09 软件体系结构基础

发表于 2024-03-07 分类于 软件工程与计算II

9.2 理解软件体系结构

9.2.1 定义

​ 软件体系结构 = { 部件 (component)，连接件 (connector)，配置 (configuration) }

  • "部件"是基本组成单位之一，承载系统主要功能，包括处理与数据。
  • “连接件”定义了部件间的交互，是连接的抽象表示。
  • “配置”定义了“部件”以及“连接件"之间的关联方式，将它们组织成系统的计算机部件与部件之间的交互。

​ 一个软件系统的体系结构规定了系统的计算部件和部件之间的交互。

​ 连接件是一个与部件平等的单位。部件与连接件是比类、模块等软件单位更高层次的抽象。

9.2.2 区分软件体系结构的抽象与实现

​ 以部件、连接件和配置为基本单位组织的模型就是软件体系结构的抽象，基本目的是描述软件系统的整体功能和组织，不涉及程序设计语言提供的各种编程机制。

​ 软件体系结构设计是先使用抽象机制完成软件系统的总体功能部署，然后再将抽象模型等价转化为实现模型。

​

9.2.3 部件

​ 部件可以分为原始 (primitive) 和复合 (composite) 两种类型。原始类型的部件可以直接被实现为相应的软件实现机制，具体实现粒度要依部件的复杂度而定。复合类型的部件则由更细粒度的部件和连接件组成。

​

9.2.4 连接件

​ 连接件也分为原始和符合两种类型。原始类型的连接件可以直接被实现为相应的软件实现机制。复合类型的连接件则由更细粒度的部件和连接件组成。

​

​ 连接件的软件实现可以分为隐式和显式两种。隐式连接件不需要专门开发，附属在部件的实现之上，部件实现完成，连接件实现自然也就完成。显示类型机制通常需要进行一些复杂的逻辑处理，需要开发者进行专门的实现。

9.2.5 配置

​ 配置是一种机制，专门将部件和连接件整合起来，构成系统的整体结构，达到系统的设计目标。

​ 体系结构语言 (Architecture Description Language, ADL)，用于描述软件体系结构的形式化模型语言。

​ 利用配置连接相互独立的部件和连接件，由如下好处：

  • 可以实现部件和连接件的一次定义，多次使用；
  • 在具体交互中，参与的部件不再固定，可以随时发生变化；
  • 对于具体部件而言，参与的交互也不再固定，可以随时参与或退出一个交互。

​

9.3 体系结构风格初步

耦合性（英语：Coupling）或称耦合力或耦合度，是一种软件度量，是指一程序中，模块及模块之间信息或参数依赖的程度。

内聚性（英语：Cohesion）是一个和耦合性相对的概念，一般而言低耦合性代表高内聚性，反之亦然。耦合性和内聚性都是由提出结构化设计概念的赖瑞·康斯坦丁所提出[1]。低耦合性是结构良好程序的特性，低耦合性程序的可读性及可维护性会比较好。

9.3.1 主程序 / 子程序
  1. 简要描述

    层次结构。

    设计决策与约束有：

    • 基于声明 - 使用（程序调用）关系建立连接件，以层次分解方式建立系统部件构成层次结构。
    • 每个上层部件可以“使用”下层部件。
    • 系统应该是单线程执行。
    • 子程序只能够通过上层转移来获得控制权。

  2. 实现

    主要实现机制是模块实现，主程序 / 子程序风格是基于部件与连接件建立的高层结构，它的部件不同于程序，而是更加粗粒度的模块。

  3. 效果

    优点：

    • 流程清晰，易于理解。
    • 强控制性。

    缺点：

    • 强耦合，难以修改和复用。
    • 限制了各部件间的数据交互，可能使得不同部件使用隐含的共享数据交流，产生不必要的公共耦合，破坏“正确性”控制能力。

  4. 应用

    主要用于能够将系统功能依层次分解为多个顺序执行步骤的系统。

9.3.2 面向对象式

  1. 简要描述

    该风格将系统组织为多个独立的对象，每个对象封装其内部的数据，并基于数据对外提供服务。不同对象之间通过协作机制共同完成系统任务。

    设计决策与约束：

    • 使用信息内聚达到标准为系统建立对象部件。每个对象对外提供服务接口，并隐藏内部数据表示。
    • 基于方法调用机制建立连接件，将对象连接起来。
    • 每个对象负责维护其自身数据的一致性和完整性，并以此为基础对外提供“正确”的服务。
    • 每个对象都是一个自治单位，不同对象之间是平级的，没有主次、层次、分解等关系。

  2. 实现

    它的“对象”是部件，属于高层结构的元素，但不同于面向对象分析方法中的“对象”实体。主要实现机制是模块实现。

  3. 效果

    优点：

    • 内部实现的可修改性。
    • 易开发、易理解、易复用的结构组织。

    缺点：

    • 接口的耦合性。
    • 标识 (identity) 的耦合性，一个对象要与其它对象交互，就必须知道其它对象的标识。
    • 副作用：对象的重入 (reentry) 问题：A.f() 调用了 B.p()，B.p() 又调用了 A.q()，就有可能导致 A.q() 失败。

  4. 应用

    适用于哪些能够基于数据信息分解和组织的系统软件。

9.3.3 分层

  1. 简要描述

    分层风格根据不同的抽象层次，将系统组织为层次式结构。每个层次被建立为一个部件，不同部件之间通常用程序调用的方式连接。

    决策与约束有：

    • 最底层到最高层，部件的抽象层次逐渐提升，
    • 两个层次之间的连接要遵守特定的交互协议，该协议是成熟、稳定和标准化的。
    • 跨层次的连接是禁止的。
    • 逆向连接是禁止的。

  2. 实现

    分层风格通常是不限粒度的模块实现。每个层次部件可以表示为一个包，包内含有它的所有实现模块。

  3. 效果

    优点：

    • 设计机制清晰，易于理解。
    • 支持并行开发。
    • 更好的可复用性与内部可修改性。

    缺点：

    • 交互协议难以修改。
    • 性能损失。禁止跨层调用，带来冗余调用处理，致使性能损失。
    • 难以确定层次数量和粒度。

  4. 应用

    适用于如下特点系统：

    • 主要功能是能够在不同抽象层次上进行任务分解的复杂处理。
    • 能够建立不同抽象层次之间的稳定交互协议。
    • 没有很高的实时性需求，能够容忍稍许的延迟。

    如 TCP/IP 网络模型，ISO 网络通信模型。

9.3.4 MVC

  1. 名称

    模型 - 视图 - 控制 (Model-View-Control, MVC) 风格。

  2. 简要描述

    以程序调用为连接件，将系统功能组织为模型、视图和控制三个部件。

    模型封装了系统数据和状态信息，实现业务逻辑，对外提供数据服务和执行业务逻辑。

    视图封装了用户交互，提供业务展现，接收用户行为。

    控制封装了系统控制逻辑，根据用户行为调用需要执行的业务逻辑和数据更新，并且根据执行后的系统状态决定后续的业务展现。

    设计决策与约束有：

    • 模型、视图、控制是关于业务逻辑、表现和控制的三种不同内容抽象。
    • 如果视图需要持续地显示某个数据状态，那么它首席需要在模型中注册1对该数据的兴趣。
    • 视图只能使用模型的数据查询服务，只有控制部件可以调用可能修改模型状态的程序。
    • 用户行为虽由视图发起，但必须转交给控制部件处理。
    • 模型部件相对独立，既不依赖于视图，也不依赖于控制。

  3. 实现

    需要为模型、视图、控制每个部件实例建立模块实现，各模块存在导入/导出关系，程序调用连接件不需要显式实现。

  4. 效果

    优点：

    • 易开发性。
    • 视图和控制的可修改性。
    • 适宜于网络系统开发的特征。

    缺点：

    • 复杂性。
    • 模型修改困难。

  5. 应用

    主要用于网络系统开发。

软件工程与计算II-0x08 软件设计基础

发表于 2024-03-04 分类于 软件工程与计算II

8.1 软件设计思想的发展

​ 20 世纪 60 年代中后期到 70 年代前中期，组织结构从语句上升为函数。

​ 20 世纪 70 年代中后期到 90 年代，从函数级别转向模块级别的“软件设计”：首先进行概要设计，根据功能设计软件的整体模块结构；其次进行详细设计，建立模块的层次化分解，定义高质量、可实现的细化模块，并设计各细化模块内部的程序结构。

  • 软件设计是独立于软件实现的活动，需要独立的方法、模型和工具。
  • 结构化设计方法的提出。数据流图和结构图是结构化设计方法的核心技术。
  • 面向对象设计逐渐发展起来。抽象数据类型、信息隐藏、封装、多态、继承等思想提出。

​ 20 世纪 90 年代之后，软件设计也进入“大规模软件设计”年代。

  • 复用思想越来越得到重视，包括产品线、框架、设计模式等“基于复用的设计”和“为复用而设计”的思想。

8.2 软件设计的核心思想

​ 分解 (decomposition) 是横向上将系统分割为几个相对简单的子系统 (subsystem) 以及各子系统间的关系。抽象 (abstraction) 则是在纵向上聚焦各子系统的接口。这里的接口 (interface) 和实现 (implementation) 相对。

​

​ 分解和抽象都是具有层次性的。

​

8.3 理解软件设计

8.3.1 设计与软件设计
8.3.2 工程设计与艺术设计

​ 从工程设计角度看，软件设计要：时刻保持以用户为中心，为其建造有用的软件产品；将设计知识科学化、系统化；能够进行设计决策与折中，解决设计过程中出现的不确定性、信息不充分、要求冲突等复杂情况。

​ 艺术性要求设计师学会：发散性思维与创新；与用户共情；进行相关因素的评价和平衡；构思与想象。

8.3.3 理性主义和经验主义

​ 理性主义更看重设计的工程性。

​ 经验主义者则在重视工程性同时，也强调艺术性，要求给软件设计过程框架添加一些灵活性以应对设计中人的因素。

8.3.4 软件设计的演化性

​ 真实的设计过程是演化与迭代的，而不是一次完成的。

​

8.3.5 软件设计的决策性

​ 软件设计通常以如下知识为决策依据：设计师自身的从业经验、类似系统的设计、参考模型、设计约定、设计原理、现有的体系结构风格和设计模式等。

​

​ 整个设计过程其实是一个设计决策链，且具有不可逆性。一旦作出某个决策，很难通过回退完全消除其影响。

​

8.3.6 软件设计的约束满足和多样性

​ 约束可以帮助设计师暂时验证解空间中多样设计方案的合理性。

​ 约束是随着设计过程深入而逐渐发现的，而不是一开始就能完全明确的；不同的设计时间点，约束会发生变化。这两个原因决定了设计过程不是线性的，而是迭代的。

8.4 软件设计的分层

​ 软件设计可分为高层设计、中层设计和低层设计。

​ 高层设计基于反映软件高层抽象的构件层次，描述系统的高层结构、关注点和决策设计。

​ 中层设计更加关注构建的模块的划分、导入 / 导出、过程之间调用关系或者类之间的协作。

​ 低层设计则深入模块和类的内部，关注数据结构、算法、类型、语句和控制结构等。

8.5 软件设计过程的主要活动

​ 分析设计出发点、建立候选方案、生成最终方案、评价。

​

8.6 软件设计的方法和模型

8.6.1 软件设计方法

  1. 结构化设计

    采用自下向上和逐步求精的思路，按照功能对系统进行分解。

  2. 面向对象设计

    利用封装、继承、多态等方法，提高软件的可扩展性和可复用性。

  3. 数据结构为中心设计

    开始于系统操纵的数据结构而不是其功能。

  4. 基于构件的设计

    重在构件的提供、开发和集成，以提高系统的可复用性。

  5. 形式化方法设计

    通过数学方法对复杂系统建模。

8.6.2 软件设计的模型

​ 静态模型是通过快照的方式对系统进行描述，通常描述状态而不是行为。

​ 动态模型通常描述系统行为和状态转移。

软件工程与计算II-0x07 需求文档化与验证

发表于 2024-03-04 分类于 软件工程与计算II

7.1 文档化原因

​ 软件开发任务可以分解为多个子任务，子任务与人员之间存在着错综复杂的关系，存在大量交流与沟通。

7.2 需求文档基础

7.2.1 需求文档的交流对象
  • 用户。
  • 项目管理者。
  • 设计人员和程序员。
  • 测试人员。
  • 文档编写人员。
  • 维护人员。
7.2.2 用例文档

​ 用例文档和需求规格说明文档是最为常见的两种需求文档，用例文档从用例角度以用例文本为主描述软件系统与外界的交互，软件需求规格说明文档则从软件产品的角度以系统级需求列表的方式描述软件系统解决方案。

​ 用例文档的基本职责是把问题域信息和需求传达给软件系统解决方案的设计者，它的书写方法和内容精确度不同于软件需求规格说明文档。

7.2.3 软件需求规格说明文档

​ 软件需求规格文档最佳内容组织方式依产品的应用领域不同而有所不同。

7.3 需求文档化要点

7.3.1 技术文档写作要点

  1. 简洁

    很少使用各种修辞手法，多是平铺直叙。尽量使用简单语句，不用复杂长句，避免使用形容词和副词。

  2. 精确

    避免歧义。

  3. 易读（查询）

    • 有效使用引言、目录、索引等能够增强文档易读性的方法。
    • 使用系统化方式组织内容信息，提供文档内容可读性。

  4. 易修改

    对于文档中必要的冗余重复信息，可以考虑使用引用：在文档中交叉引用相关各项。

7.3.2 需求书写要点

  1. 使用用户术语

    不要使用计算机术语。

  2. 可验证

    如：用户完成任何一个查询任务时鼠标点击数不能超过 5 次。

  3. 可行性

    需求可行性不仅需要考虑理论上技术实现可能性，更要考虑在限定的成本、时间和人员约束内，实现需求的可行性。

7.3.3 软件需求规格说明文档书写要点
  1. 充分利用标准的文档模板，保持所有内容位置得当。
  2. 保持文档内的需求集具有完备性和一致性。不能遗漏重要需求或必要信息。一致性则是指不同需求之间不能互相冲突。
  3. 为需求划分优先级。可以划分为高、中、低三种优先级，也可以划分为 1~10 定量优先级。

7.4 评审软件需求规格说明文档

7.4.1 需求验证与确认

​ 评审是进行需求验证与确认的主要方法，原则上每一条需求都应该评审。

7.4.2 需求评审的注意事项

  1. 重视需求评审。

  2. 需求评审的组织：

    1. 评审人员不能仅由技术人员组成，必须包括客户和用户。
    2. 在评审中使用线索。
    3. 使用需求检查列表。

软件工程与计算II-0x06 需求分析方法

发表于 2024-02-29 分类于 软件工程与计算II

6.1 需求分析基础

6.1.1 需求分析的原因

​ 需求获取中得到的信息仅仅解释了用户对软件系统的期待，还不是开发者能够立即加以实现的解决方案。需求分析的任务是：

  • 建立分析模型，达成开发者和用户对需求信息的共同理解。关键活动是建模。

  • 依据共同理解，发挥创造性，创建软件系统解决方案。

6.1.2 需求分析模型

  1. 建模

    抽象 (abstraction) 和 分解 (decomposition / partitioning) 是建模最常用的两种手段。

  2. 需求分析模型的特点及常见分析模型

    需求分析模型一方面使用了软件的内部实体，以对象、类、函数、过程、属性等作为模型的基本元素；另一方面使用问题域的概念来描述软件内部实体，使用问题域语言表现语义。

6.2 结构化分析

6.2.1 结构化分析方法

​ 结构化分析方法把现实世界描绘为数据在信息系统中的流动，以及在数据流动过程中数据向信息的转化。

​ 数据流图 (DFD) 是结构化分析方法的核心技术，表明系统的输入、处理、存储和输出，以及它们如何在一起协同工作。

​ 实体关系图 (ERD) 是另一个核心技术，用来描述系统需要存储的数据信息。

6.2.2 数据流图

  1. 基本元素

    外部实体、过程、数据流和数据存储。

  1. 语法规则

    1. 必须有输入，也必须有输出。
    2. 数据流必须和过程关联。
    3. 每个对象必须有唯一标识名称。

  2. 分层结构

    分层的 DFD 定义了三个层次类别：上下文图、0 层图和 N 层图。

    1. 上下文图：最高层次，将整个系统看作一个功能，实现系统所有功能。
    2. 0 层图：是上下文图单一过程的细节描述，是对该单一过程的第一次功能分解，需要在一个图中概况系统所有功能。
    3. 分解规则：要求 DFD 子图的输入流、输出流必须和父过程的输入流、输出流保持一致。
6.2.3 实体关系图 (Entity Relation Diagram, ERD)

​ ERD 描述数据的定义、结构和关系等特性。

​ 最小基数为 0 为 Optional，最小基数为 1 为 Mandatory。

​ 最大基数为 1 为 One， 最大基数大于 1 为 Many。

​ 实体是需要在系统中收集和存储的现实世界事物的类别描述。

​ 属性是可以对实体进行描述的特征。

​ 若某个标识可以唯一确定实体中的一个实例，则称为标识符或键 (key)。

6.3 面向对象分析

6.3.1 面向对象分析方法

​ 统一建模语言 (Unified modeling Language, UML) 是面向对象分析的主要模型技术。常见有类图、用例图、交互图（顺序图）、状态图和对象约束语言 (OCL)。

6.3.2 用例

​ 一个用例承载了所有与用户某一个目标相关的成功和失败场景的集合。

6.3.3 用例图

  1. 基本元素

    用例、参与者、关系和系统边界。

  2. 建立

    1. 进行目标分析与确定解决方向
    2. 寻找参与者
    3. 寻找用例

    1. 细化用例：用例描述了为应对一个业务事件，由一个用户发起，并在一个连续时间段内完成，可以增加业务价值的任务。

      • 不要将粒度细化过小，将用例细化为单个操作。
      • 不要将同一个业务目标细化为不同用例。
      • 不要将没有业务价值的内容作为用例。
6.3.4 用例描述
  • 围绕“交互”进行场景描述。
  • 保持“规格说明”级别，尽可能不要涉及界面、按钮、方法等软件内部构造机制。
6.3.5 概念类图（领域模型）

​ 基本元素有对象、类、链接、关联（聚合）、继承。

​ 对象有三方面内容：标识符、状态、行为。

​ 概念类会显示描述自己一些重要属性，但不是全部的详细属性，而且概念类的属性通常没有类型的约束。

​ 类之间的关系被称为关联 (association)。

​ 聚合 (aggregation)：空心菱形。标识整体与部分的关系。

​ 组合 (composition)：实心菱形。一旦一个部分属于某个整体，就无法属于其它整体，也无法单独存在。

​ 继承：三角箭头。

​ 建立概念类图：

  • 建立局部概念类图。
  • 合并，建立软件系统整体概念类图。识别候选类、确定概念类、识别关联、识别重要属性。
6.3.6 交互图（顺序图）

​ 交互图是描述对象协作的技术。

​ 顺序图：纵向是时间轴，向下延伸。横轴表示参与协作的对象。

​ 系统顺序图是将整个系统看作一个黑箱的对象而描述的简单顺序图形式，强调外部参与者和系统的交互行为，重点展示系统级事件。

6.3.7 状态图

​ 有限状态机是状态图的基础。

​ 建立状态图：

  • 确定上下文环境。
  • 识别状态。建立状态转换。
  • 补充详细信息，完善状态图。

6.4 使用需求分析方法细化和明确需求

6.4.1 细化和明确需求内容

​ 需求分析模型的建立是为了改进用例文本描述的内容，细化和明确需求，发现其中的冗余、遗漏、错误和冲突。

6.4.2 建立系统级需求

​ 需要分析完所有流程的系统顺序图，才能得到所有的刺激与响应。

软件工程与计算II-0x05 软件需求基础

发表于 2024-02-28 更新于 2024-02-29 分类于 软件工程与计算II

5.1 引言

​ 需求开发阶段主要任务是分析问题，研究问题所发生的现实世界（即问题域），寻找实现软件系统与现实世界的有效互动办法，并严描述该互动办法（即建立软件解决方案，又称软件规格说明）。

​ 软件需求开发：既需要从问题出发，分析问题域，研究解决问题所需要的互动效应；又需要关注软件系统的解决方案，建立软件规格说明，保证基于软件规格说明构造的软件产品能够解决用户问题。

​ 常见的忽略需求开发的情况有：

  • 问题广为人知或者需求非常明确。
  • 问题小而简单。

5.2 需求工程基础

5.2.1 需求工程简介

​ 需求工程三个主要任务：

  • 说明软件系统将被应用的环境及其目标，说明用来达到这些目标的软件功能，即要同时说明软件“需要做什么”和“为什么需要做”。

  • 将目标和功能反映到软件系统中，映射为可行的软件行为，并对软件行为进行准确的规格说明。

  • 妥善处理目标和功能随着时间演化的变动情况。

5.2.2 需求工程活动

​ 需求开发是专门用来处理需求的软件技术。

​ 需求获取目的是从空白开始建立最初的原始需求。研究系统将来的应用环境，确定涉众，了解现有问题，建立系统目标，获取业务细节和具体用户需求。

​ 需求分析目的是保证需求的完整性和一致性。从需求获取阶段输出的原始需求和业务细节出发，建立系统模型和系统及需求，标识并修复需求缺陷，发现并弥补遗漏需求。

​ 需求规格说明目的是将完整、一致的软件解决方案以文档的方式固定下来。

​ 需求验证的目的是保证软件需求规格说明文档的质量，要求文档内需求正确地反映用户的真实意图，并保证整个文档的一致性和完整性。

​ 需求管理是对需求开发所建立的需求基线的管理，在基线完成后正式开始，并在需求开发结束之后持续存在。

5.2.3 需求获取

​ 任务有两方面。

  1. 目标分析
  • 根据问题确定目标。发现问题比发现目标简单一些，问题反面就是目标。
  • 通过分析利害关系人确定目标。
  1. 用户需求获取
  • 面谈
  • 集体获取方法
  • 头脑风暴
  • 原型：建立一个有形的制品来增进用户与需求工程师之间的交流。
5.2.4 需求分析
  1. 边界分析

​ 系统边界定义了项目范围，边界之内定义的是系统需要对外提供的功能，边界之外标识的是对系统有功能要求的外部实体或者对系统有所限制的环境要素。

​ 系统用例图和上下文图常用来定义系统边界。

  1. 需求建模

​ 模型能够在有限空间内表述更加严谨、准确和高密度的信息，建模是需求分析中最为基础和重要的一项任务。

​ 常用技术包括数据流图、实体关系图、状态转换图、类图等半形式化建模技术。更严格的领域也会应用 Z 模型等更加严格要求的形式化技术。

5.2.5 需求规格说明
  1. 定制文档模板
  2. 编写文档
5.2.6 需求验证

​ 需求规格说明文档至少应满足以下标准：

  • 文档内每条需求都正确、准确地反映用户意图。
  • 文档记录的需求集在整体上具有完整性和一致性。
  • 文档的组织方式和需求的书写方式具有可读性和可修改性。
5.2.7 需求管理

​ 需求管理会进行变更控制，纳入和实现合理的变更请求，拒绝不合理的变更请求，控制变更的成本和影响范围。

5.3 需求基础

5.3.1 需求

​ 需求通常被表述为“系统应该……”、“在……时，系统应该……”。如：

  • 系统应该允许顾客退回已经购买的产品。
5.3.2 需求的层次性

  1. 业务需求

    描述了组织为什么要开发系统。

    高层次解决方案与系统特性说明了系统为用户提供的各项功能，限定系统范围，帮助用户和开发者确定系统边界。

  2. 用户需求

    用户需求是执行工作的用户对系统所能完成具体任务的期望，描述了系统能够帮助用户做些什么。

  3. 系统级需求

    系统级需求是用户对系统行为的期望，每个系统及需求反映了一次外界与系统的交互行为，或是系统的一个细节。

    系统级需求可以直接映射为系统行为，定义了系统中需要实现的功能，描述了开发人员需要实现什么。

5.3.3 结合层次性的需求开发

​ 软件需求的开发从业务需求开始，它具有明显的目的性，比较容易获取与确认。

​ 业务需求影响着用户需求的获取。

​ 很多情况下得到用户需求后即可结束需求开发，进入设计阶段。但系统比较复杂的情况下，还需继续细化用户需求，处理为等价的系统级需求。

5.3.4 区分需求、问题域与规格说明

​ 需求是一种期望，既源于现实又高于现实。

​ 问题域是对现实世界运行规律的一种反映，是需求的生产地，也是需求的解决地。

​ 规格说明是软件产品的方案描述，以软件产品的运行机制为主要内容。不是需求但实现需求，不是问题域但需要与问题域互动。

5.4 需求分类

5.4.1 需求谱系
5.4.2 软件需求分类
  • 功能需求
  • 性能需求
  • 质量属性
  • 对外接口
  • 约束
  • 数据需求等其它需求

  1. 功能需求：

    最常见、最主要、最重要的需求，也是最复杂的需求。

    功能需求是一个软件产品能够解决用户问题和产生价值的基础，是整个软件开发工作的基础。

  2. 性能需求：

    速度、容量、吞吐量、负载、实时性。

  3. 质量属性：

    一方面质量属性是隐式的；另一方面在后续开发中，质量属性需求会极大影响软件体系结构设计。常见质量属性有：可靠性、可用性、安全性、可维护性、可移植性。

  4. 对外接口：

    对外接口部分通常被封装起来，防止接口的另一方发生变化时，尽可能少地影响软件系统内部。

    重要的用户界面接口需要提供界面原型。

    对系统间的软硬件接口和通信接口需说明以下内容：

    • 接口的用途
    • 接口的输入、输出
    • 数据格式
    • 命令格式
    • 异常处理要求

  5. 约束：

    约束是在总体上限制开发人员设计和构建系统时的选择范围。常见约束有三类：

    • 系统开发及运行的环境。
    • 问题域内的相关标准，包括法律法规、行业规定、企业规章等。
    • 商业原则。

  6. 数据需求：、

    数据需求是需要在数据库、文件或者其他介质中存储的数据描述，通常包含：

    • 各个功能使用的数据信息；
    • 使用频率；
    • 可访问性要求；
    • 数据实体及其关系；
    • 完整性约束；
    • 数据保持要求。

软件工程与计算II-0x04 项目管理基础

4.1 项目和项目管理

​ 项目是具有如下特征的一系列活动或任务：

  • 具有一个明确的目标；
  • 有限定的开始和结束日期；
  • 有成本限制；
  • 消耗人力或非人力资源；
  • 多工种合作。

​ 项目的核心是计划，它建立计划，并跟踪、监督和保证计划的正确执行。计划研究内容包括：项目需要的资源、项目中需要执行的活动（生命周期模型和过程模型），以及需要产生的交付产品（最终产品和各种必要的中间产品）。

​ 围绕项目计划进行的管理活动是项目管理，目标包括：

  • 在限定时间内；
  • 在一定成本内；
  • 在要求的质量水平上；
  • 高效使用资源；
  • 获得用户认可。

​ 软件项目管理包括项目启动、项目计划、项目执行、项目跟踪与控制和项目收尾 5 个过程组，需要执行计划制定、团队管理、成本控制、质量保障、度量、过程管理、进度跟踪与控制、风险管理、配置管理等各种具体活动。

4.2 团队组织与管理

4.2.1 团队特征
  • 具备共同的目标。
  • 共担责任。
  • 技能互补。
  • 小规模团体。
  • 明确的结构。
4.2.2 团队结构
  1. 主程序员团队

​ 团队中有一名能力出色的成员指定为主程序员，负责领导团队完成任务。主程负责完成总体构思和设计，然后分配任务给其他团队成员，并监督、验收和整合其他成员的工作完成情况。

​ 优点在于高效率。

​ 缺点在于如果项目复杂或主程能力不足，则主程将称为瓶颈。

  1. 民主团队

​ 没有集中的交流点，每个团队都能发挥能动性，取得更高士气和工作成就感。工作效率不如主程序员团队，花费在统一思想和解决冲突上的代价不可小视，特别需要防止陷入混乱，

​ 敏捷过程使用民主团队方式。

  1. 开放团队

​ 开放团队为了创新而存在，采用黑箱管理方式，内部交流路径对管理者不可见。

​ 团队可以按照自己适合的方式自我管理，激励成员主动性，提高创新能力。

​ 问题在于项目进展没有可视度，当然创新活动本身缺乏可视度。

4.2.3 团队建设
  1. 建立团队章程

​ 团队章目的：统一团队成员的目标、绩效和方法，指导团队管理工作的进行。内容包括：

  • 团队的目标。
  • 共同的追求，这个追求是超出项目之外的，如追求敏捷理念、开源理念等。
  • 团队结构和角色分工。
  • 团队的任务、活动与绩效。
  • 团队规则与约束。
  1. 持续成功

​ 可以是项目整体成功，也可以是项目阶段性成功，还可以是其他无关项目的团队活动的成功。原因在于：

  • 积累团队信心。
  • 建立团队成员间的信任。
  • 激励团队士气。

​ 常见做法有：项目开始和空闲时开展团队活动；项目中设立小里程碑。

  1. 和谐沟通

​ 首要原则是建立开放的环境，鼓励交流。

​ 需要有制度保障，需要有定期沟通的只读，保证团队成员间正确、及时的沟通。

​ 交流是双向的，团队成员（工作进展）与管理者（整体状态）间应积极沟通。

​ 沟通分正式和非正式沟通。必要的书面交流是需要的。

  1. 避免团队杀手

  • 防范式管理。

  • 官僚主义。

  • 地理分散。

  • 时间分割。团队成员参与多个项目，为每个项目分割自己一部分时间，则难以形成胶冻团队。

  • 产品质量降低。

  • 虚假的最后期限。明显不可能完成任务的期限。

  • 小圈子控制。

4.3 软件质量保障

4.3.1 软件质量

​ 对软件质量要求可能是显式的，也可能是隐式的。“隐式的”指用户在软件系统创建之前无法清晰表达却可以在软件系统投入使用之后要求补充的条件。软件系统为满足显示及隐式要求而具备的要素称为质量。

​ 质量属性：人们会选用系统的某些质量要素进行量化处理，建立质量特征，这些特征称为质量属性。

​ 质量模型：为了根据质量属性描述和评价系统的整体质量，人们从很多质量属性定义中选择了一些能够互相配合、联系的特征集，称为质量模型。

4.3.2 质量保障

​ 一方面，软件开发过程不可见；另一方面，越晚发现缺陷，修复成本越高。因此对软件质量保障活动要贯穿整个开发过程独立、持续进行。

​ 项目启动时需进行质量保障计划，明确需要执行的质量保障活动，指明保障活动的时机与方法。

​ 开发活动达里程碑时，及时根据计划进行质量验证。验证方法有评审、测试和质量度量三种。

4.3.3 评审 (review)

​ 又称同级评审，由作者之外的其他人来检查产品问题，是静态分析手段。

  • 规划阶段：制定审查计划。
  • 总体部署阶段：向所有审查任意描述审查材料内容、目标及假设，分发文档。
  • 准备阶段：审查人员各自独立执行检查任务。
  • 审查会议：通过会议讨论，识别、确认、分类发现的错误。
  • 返工阶段：修改发现的缺陷。
  • 跟踪阶段：确认所有问题得到解决，所有错误得到修正。
4.3.4 质量度量

​ 度量 (metric) 是软件产品在特定属性上的量化测量程度。

4.4 软件配置管理

4.4.1 动机

​ 复杂软件系统中，产生制品数量众多，以至于开发者需要维护一个清单才能清楚项目状态。这些制品会在开发过程中发生变化。

​ 某个制品发生变化带来最大挑战就是如何确保使用者能够得到最新制品，避免开发协同问题。

​ 软件配置管理将软件开发重要制品及其变更纳入管理和监控，保证了不影响协同活动情况下处理变更。

4.4.2 配置项

​ 需要进行配置管理的软件开发制品称为配置项。

4.4.3 基线

​ 纳入配置管理和执行变更控制的制品称为该配置项的基线 (baseline)。

4.4.4 配置管理活动
  • 表示配置项。
  • 版本管理：版本更新时i，保留而不是删除旧记录，以便回滚。
  • 变更控制。
  • 配置审计：确定一个项目满足需求的功能和物理特征的程度。
  • 状态报告：对动态演化着的配置项信息及度量取快照。
  • 软件发布管理。
4.4.5 变更控制

​ 对变化的评估、协调、批或拒绝准、实现和验证。

​ 变更控制委员会 (Change Control Board, CCB) 根据变更评估的信息作出批准或拒绝变化的决定。
